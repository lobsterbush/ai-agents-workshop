\title{Partisan Bias in Evaluating Political Scandals: A Survey Experiment}

\begin{abstract}
Do partisans apply different standards when evaluating misconduct 
by co-partisan versus opposing-party politicians? We present a 
pre-registered survey experiment (N = 2,000) in which respondents 
evaluate a state legislator accused of financial misconduct. We 
randomize the party affiliation of the accused politician while 
holding the scandal constant.

We find that respondents rate co-partisan politicians as significantly 
less culpable than opposing-party politicians accused of identical 
misconduct. This effect is strongest among strong partisans and 
persists after controlling for demographic characteristics.
\end{abstract}

## 1. Data collection
Prolific, N = 2,000 US adults.

## 2. Hypotheses
H1 (In-party bias): Respondents will rate politicians from their 
own party as less culpable than opposing-party politicians.

H2 (Asymmetry): Effect stronger among strong partisans.

## 3. Dependent variables
- Culpability (7-point: "not at all" to "completely")
- Appropriate punishment (5-point scale)
- Vote likelihood (if Torres ran for re-election)

## 4. Analyses
OLS with robust SEs. Interaction: politician_party × respondent_party.
Controls: age, gender, education, political interest.

Output

01_analysis.R

library(tidyverse)
library(estimatr)
library(modelsummary)

df <- read_csv("data/scandal_experiment.csv") |>
  mutate(
    co_partisan = (respondent_party == politician_party),
    strong_partisan = abs(party_id_7) >= 3
  )

# H1: In-party bias
m1 <- lm_robust(culpability ~ co_partisan + age + female + college,
                data = df, se_type = "HC2")

# H2: Interaction with partisan strength
m2 <- lm_robust(culpability ~ co_partisan * strong_partisan + 
                age + female + college, data = df, se_type = "HC2")

modelsummary(list("H1" = m1, "H2" = m2), 
             output = "output/tables/main_results.tex")

Part 1

How Agents Work

Agents have tools

They can interact with your computer the same way you do.

  • File system — read, write, organize, delete
  • Terminal — run commands, install packages, execute scripts
  • Code execution — run R, Python, Stata; see errors; fix them
  • Web access — search documentation, fetch data, verify information

The loop

Agents observe each result and decide what to do next.

  1. Observe current state
  2. Plan next action
  3. Execute action
  4. Check if it worked
  5. Repeat until done

This is why they recover from errors. They see what went wrong and try again.

Part 3

Where They Fail

Known failure modes

These tools are powerful but not infallible.

  • Hallucinated citations — they invent plausible-sounding references that don't exist
  • Wrong statistics — numbers can be subtly incorrect; always verify
  • Overfit to prompt — they do exactly what you ask, even if it's the wrong thing to ask
  • Confident errors — mistakes are presented with the same confidence as correct output

The verification principle

Treat agent output like a research assistant's first draft.

  • Run the code yourself
  • Spot-check calculations against known values
  • Verify every citation exists (Google Scholar, library)
  • Review the logic, not just the output
  • Check edge cases and assumptions

Implications

The question isn't whether

These tools are here. They're getting better rapidly. The question is how we use them responsibly as a scholarly community.

Discussion
